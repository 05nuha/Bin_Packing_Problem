① What is Bin Packing?

Imagine you are moving house. You have boxes of different sizes and you want to pack everything into the fewest moving trucks possible. Each truck has a fixed capacity. That is Bin Packing.

Formally: you have n items, each with a size between 0 and 1. Bins each have capacity 1. Your goal — pack all items into the fewest bins possible.

Items: [0.4, 0.6, 0.3, 0.7]  |  Capacity = 1.0
Bin 1: 0.7 + 0.3 = 1.0 ✓ (perfectly full)
Bin 2: 0.6 + 0.4 = 1.0 ✓ (perfectly full)
2 bins — zero waste.

Why is this hard? With 10 items and 4 bins there are 4¹⁰ = over a million possible assignments. With 20 items it is billions. You cannot try every combination — this is why Bin Packing is NP-complete.

💡Real uses: Shipping containers, cutting fabric rolls, cloud server allocation, compiler memory allocation.

② The Four Algorithms

This tool implements four approaches to Bin Packing — ranging from exact-but-slow to approximate-but-fast.

📌What does O() mean? It tells you how fast an algorithm grows as input size n increases. O(n log n) is fast — barely slower than reading the input. O(n²) is moderate — doubles the work when n doubles. O(kⁿ) is exponential — adding one item multiplies the work by k. For n=20 that means billions of operations.
Brute Force — tries every possible assignment. Finds the true optimal. Time: O(kⁿ) — exponential. Only practical for n ≤ 8.
First-Fit (FF) — for each item in original order, place in the first bin that fits. Open a new bin if nothing fits. Time: O(n²).
Best-Fit (BF) — same idea but choose the bin with the least remaining space that still fits. Packs tighter locally. Time: O(n²).
First-Fit Decreasing (FFD) — sort items largest-first, then apply First-Fit. The best practical algorithm. Guarantee: ≤ (11/9)·OPT bins. Time: O(n log n).
💡Brute Force is the gold standard — it always finds optimal. But it is too slow for large inputs. FFD is the practical choice: fast and provably close to optimal.

③ Worked Example — FFD Step by Step

Items: 0.7, 0.5, 0.4, 0.3, 0.2, 0.1. Capacity = 1.0. Already sorted largest-first.

Step 1: Item 0.7 — no bins open. Open Bin 1. Place 0.7. Bin 1: [0.7] free: 0.3
Step 2: Item 0.5 — Bin 1 has 0.3 free, 0.5 does not fit. Open Bin 2. Bin 2: [0.5] free: 0.5
Step 3: Item 0.4 — Bin 1: 0.3 free (no). Bin 2: 0.5 free (yes!). Place in Bin 2. Bin 2: [0.5, 0.4] free: 0.1
Step 4: Item 0.3 — Bin 1: 0.3 free (yes!). Place in Bin 1. Bin 1: [0.7, 0.3] free: 0.0 FULL
Step 5: Item 0.2 — Bin 1 full. Bin 2: 0.1 free (no). Open Bin 3. Bin 3: [0.2] free: 0.8
Step 6: Item 0.1 — Bin 2: 0.1 free (yes!). Place in Bin 2. Bin 2: [0.5, 0.4, 0.1] free: 0.0 FULL
Result: 3 bins. Bin 1 and Bin 2 perfectly full. Bin 3 has 0.8 free.

Key observation: The largest items (0.7, 0.5) went first and claimed space. The smaller items (0.3, 0.1) filled the remaining gaps perfectly. That is the power of sorting first.

④ Why Sorting First Makes FFD Better

The core insight: Large items are constraints — they are hard to place. Small items are flexible — they can fill any gap. Place constraints first, fill gaps with flexible pieces.

Without sorting, small items arrive first and fill bins in awkward ways, leaving gaps that large items cannot fit into. Extra bins open unnecessarily.

Exchange Argument — Why Sorting is Provably Correct
Lemma — Exchange Argument for FFD

Claim: Let σ be any ordering of items. Let σ' be σ with two adjacent items i and j swapped, where item i is smaller than item j. Then FFD on σ' uses no more bins than FFD on σ.

Base Case — one swap never opens an extra bin

Before swap: Small item i goes first, leaving gap g = 1 − s(i) in its bin. Large item j arrives next.

Case A — j already fits (s(j) ≤ g): Both orderings place j in the same bin. The swap changes nothing — result is identical (neutral). ✓

Case B — j doesn't fit (s(j) > g): Before swap, j is forced into a new bin. After swap, j goes first leaving gap g' = 1 − s(j). Since s(i) < s(j), we have s(i) ≤ 1 − s(j) = g'. So i fits in the same bin as j — no new bin needed. The swap saves a bin. ✓

In both cases, swapping never worsens the result. □

Inductive Step — k swaps, then one more

Assume after k adjacent swaps the first k positions are in decreasing order and the bin count has not increased. The (k+1)th swap corrects one more out-of-order pair — by the base case, this swap does not open a new bin. So after k+1 swaps the result is still no worse. ✓

Termination — reaches sorted order in finite steps

Any ordering of n items has at most n(n−1)/2 pairs that are out of order. Each adjacent swap fixes exactly one pair. After at most n(n−1)/2 swaps the sequence is fully sorted in decreasing order — and the bin count is no worse than any other ordering. □

In plain English: Whenever a small item comes before a large item, swapping them never opens an extra bin — the large item claims space first, and the small item always fits in the remaining gap. Since every possible swap is safe, the fully sorted (largest-first) order is provably optimal among all orderings. This is why FFD sorts first.

⑤ NP-Completeness in Plain English

P = problems solvable fast (polynomial time). Sorting, shortest path, searching — all in P.

NP = problems where solutions are checkable fast, even if you cannot find them fast. Bin Packing is in NP — given a proposed packing, you can check it is valid in O(n). But finding the best packing takes exponential time.

NP-complete = the hardest problems in NP. If you solve one in polynomial time, you solve all of them. Bin Packing, Sudoku, Travelling Salesman, Tetris — all NP-complete.

What this means: Since Bin Packing is NP-complete, we use approximation algorithms — fast algorithms that are provably close to optimal. FFD is the classic example.

💡NP does NOT mean impossible. It means no polynomial-time exact algorithm is known. We can approximate with FFD and solve small instances exactly with Brute Force.

⑥ The Approximation Ratio — (11/9)·OPT

Since Bin Packing is NP-complete, we can't solve it exactly in polynomial time. So FFD makes a trade-off — it runs fast but accepts being slightly above optimal. The approximation ratio tells you exactly how far off it can be.

FFD's guarantee: FFD uses at most (11/9)·OPT + 6/9 bins.

What this means step by step:
· means multiply. OPT = the true minimum bins any algorithm could use.
11/9 = 1.222... so FFD uses at most 1.22 × OPT bins.
Example: if OPT = 9 → FFD uses at most (11/9)×9 + 6/9 = 11.67 → at most 11 bins.
Never more than ~22% above optimal. Always. Guaranteed.

Counterexample — FFD is Not Always Exact

FFD is an approximation — close but not always optimal. Here is a concrete case where FFD uses one extra bin:

Items: {0.4, 0.4, 0.4, 0.3, 0.3, 0.3, 0.3, 0.3, 0.3}
FFD sorts → [0.4, 0.4, 0.4, 0.3, 0.3, 0.3, 0.3, 0.3, 0.3]
Bin 1: 0.4 in (0.6 left). 0.4 in (0.2 left). Next is 0.4 → 0.4 > 0.2, won't fit!
Bin 2: open new. 0.4 in (0.6 left). 0.3 in (0.3 left). 0.3 in (0.0 left). Bin 2 FULL.
Bin 3: 0.3 in (0.7 left). 0.3 in (0.4 left). 0.3 in (0.1 left). Bin 3 almost full.
Bin 4: last 0.3 → Bin 1 has 0.2 left (0.3 won't fit). Bin 3 has 0.1 left (won't fit). Open Bin 4.
FFD result: 4 bins.
OPT: 3 bins — each bin gets one 0.4 + two 0.3s = 0.4+0.3+0.3 = 1.0. Perfect.
Guarantee check: 4/3 = 1.33 ≤ (11/9)×3 = 3.67. ✓ Still within guarantee.

⑦ Common Mistakes — Don't Fall For These

✗ "FFD always gives the optimal answer."
Wrong. FFD is an approximation — it guarantees at most (11/9)·OPT but can exceed optimal. Load FFD vs OPT preset to see a counterexample (FFD=4, OPT=3).
✗ "NP-complete means the problem is impossible."
Wrong. NP-complete means no polynomial-time exact algorithm is known. We can still approximate (FFD) or solve small instances exactly (Brute Force).
✗ "Best-Fit is always better than First-Fit."
Wrong. Both have the same worst-case guarantee (17/10)·OPT. Best-Fit packs tighter locally but neither sorts first — so both can be beaten by FFD.
✗ "Sorting makes FFD optimal."
Wrong. Sorting improves FFD significantly but the counterexample shows it can still use more bins than optimal.

Why FFD Usually Wins

Sorting largest-first means big items claim space cleanly. Small items then fill the remaining gaps perfectly. The sort step costs O(n log n) but saves many bin openings.

Why First-Fit Can Lose

Without sorting, large items arriving late find bins already partially filled with small items — in the wrong sizes to combine. Extra bins open unnecessarily.

Best-Fit vs FFD

Best-Fit packs tighter locally (least wasted space per bin) but doesn't sort first. On sorted input both behave similarly. FFD's sort step gives it a stronger approximation guarantee.

Why Best-Fit doesn't achieve (11/9)·OPT: Best-Fit's greedy choice — always fill the tightest bin — can backfire. Tighter local packing sometimes creates many bins with awkward small gaps that later items can't fill. Without sorting, large items arrive late and find no bin large enough. The exchange argument only proves sorted-decreasing is optimal for First-Fit placement rules — Best-Fit's different placement logic is not covered by the same proof.

When Brute Force Matters

Brute Force gives the provably minimum bin count. For small n (≤8) it's feasible. For large n it's exponential — proving why FFD's approximation is the practical solution.

🧠 What Does NP-Complete Mean?

P = problems solvable fast (polynomial time). Sorting, shortest path, searching — all in P.

NP = problems where a proposed solution can be checked fast, even if it can't be found fast. Bin Packing is in NP — if someone hands you a packing, you can verify it is valid in O(n).

NP-complete = the hardest problems in NP. They're all secretly connected — solving one fast would solve ALL of NP. Bin Packing, Sudoku, Tetris, Travelling Salesman — all NP-complete.

To prove a problem X is NP-complete you need exactly two things:

① Show X is in NP — describe a certificate and show it can be verified in polynomial time.
② Show X is NP-hard — take a known NP-complete problem and reduce it to X in polynomial time.
💡NP does NOT mean impossible. It means no polynomial-time exact algorithm is known. We can still approximate with FFD — and that's exactly why approximation algorithms matter.

Step 1 — Bin Packing ∈ NP

We need to show that if someone gives us a proposed packing, we can check it quickly.

The certificate (proposed solution): an assignment of every item to a bin.
Example: Item 1 → Bin 2, Item 2 → Bin 1, Item 3 → Bin 1, ...

The verifier checks four things, all in O(n):

① Every item assigned: scan the certificate, confirm every item has exactly one bin label. Cost: O(n).
→ Makes sure nothing was left out and nothing was put in two bins at once.
② No bin exceeds capacity: for each bin, add up the sizes of all items assigned to it, check the total is ≤ 1. Cost: O(n).
→ Capacity = 1. If any bin's items add up to more than 1, the packing is invalid.
③ All item sizes valid: confirm every item size sᵢ is a number strictly between 0 and 1. Cost: O(n).
→ sᵢ is just the size of item i — for example 0.4 or 0.7. The problem requires sizes in (0,1] so we check no item is 0 or larger than 1.
④ Total bins used ≤ k: count how many distinct bin numbers appear in the certificate, verify that count is ≤ k. Cost: O(n).
→ k is the bin limit we were given as input. If the packing uses 5 bins but k=3, it fails.
Total verification cost: O(n) — polynomial. ✓

The verifier never tries to find a good packing — it only checks the given one. Finding is hard. Checking is easy. That's NP.

💡Think of Sudoku — finding the solution is hard, but checking someone else's solution takes seconds. Same idea here.

Step 2 — Reduce from Partition

What does "reduce" mean? We transform Partition — already known NP-complete — into Bin Packing, preserving the YES/NO answer. We write: Partition ≤ₚ Bin Packing.

The ≤ₚ symbol means "reduces to in polynomial time" — if you could solve Bin Packing fast, you could solve Partition fast too. Since Partition is hard, Bin Packing must be at least as hard.

⚠ Why Partition? Why not another NP-complete problem?

NP-completeness proofs form a chain — each proof builds on something already established independently:

Cook (1971) — proved 3-SAT NP-complete from scratch (the foundation)
↓ reduced from 3-SAT
Karp (1972) — proved Partition NP-complete
↓ reduced from Partition ← we are here
Us — proving Bin Packing NP-complete

We can only reduce FROM a problem already proved NP-complete before ours. Partition was proved in 1972 — independently, before Bin Packing. So we can stand on it as a given fact and cite Karp 1972 without reproving it.

We cannot use Bin Packing to prove Partition — that would be circular: "Partition is hard because Bin Packing is hard... and Bin Packing is hard because Partition is hard." Each link in the chain must point downward only.

The Partition problem: Given S = {a₁,…,aₙ} with total sum 2T, can S be split into two subsets each summing to T?

Example: S = {3,3,2,2}, total=10, T=5. Split into {3,2} and {3,2} — both sum to 5. YES.

The construction (f):

① Compute T = (Σaᵢ)/2 — half the total sum.
② Normalize: sᵢ = aᵢ/T for each element. Now every sᵢ ∈ (0,1].
③ Set k = 2 bins of capacity 1.
④ Ask: do all normalized items fit in 2 bins?

This runs in O(n) — one division per element. Polynomial. ✓

Input: S = {3,3,2,2}   T = 5
Normalize: 3/5=0.6   3/5=0.6   2/5=0.4   2/5=0.4
Bin Packing: items={0.6,0.6,0.4,0.4}, k=2, capacity=1
💡 Why do we need BOTH directions?

The reduction only works if Partition and Bin Packing have the exact same YES/NO answer on corresponding inputs. One direction alone is not enough:

Forward alone (Partition YES → Bin Packing YES): Only proves that if Partition has a solution, so does Bin Packing. But what if Bin Packing says YES on inputs where Partition says NO? Then they're not truly equivalent — Bin Packing could be easier than Partition. Hardness doesn't transfer.
Backward alone (Bin Packing YES → Partition YES): Only proves that any Bin Packing solution gives a Partition solution. But what if Partition has a solution that doesn't correspond to any valid packing? Then the translation misses cases.
Both together: Partition = YES ⟺ Bin Packing = YES. The two problems are completely equivalent — same answer, always. This means Bin Packing is at least as hard as Partition. Since Partition is NP-complete and Bin Packing ∈ NP, Bin Packing is NP-complete.
Theorem — Correctness of the Reduction

x ∈ Partition  ⟺  f(x) ∈ Bin Packing

Proof: By Lemma 1 (forward direction, next slide) and Lemma 2 (backward direction, slide after).
f runs in O(n) — one division per element. Therefore Partition ≤ₚ Bin Packing.
Since Partition is NP-complete and Bin Packing ∈ NP, Bin Packing is NP-complete. □

x ∈ Partition = "x is a YES instance of Partition"  |  f(x) ∈ Bin Packing = "the translated instance is also YES"  |  ⟺ = "if and only if" — both directions must hold

Edge Cases — The Proof Handles All Inputs

A rigorous proof must work on all valid inputs, not just typical ones.

Item of size exactly 1

Must go in its own bin. Verifier correctly flags any bin where sharing would push total above 1. Proof holds unchanged.

Single item (n=1)

Trivially fits in 1 bin. But n=1 is not a valid Partition instance — you need at least 2 elements with sum=2T, T>0. Reduction is not applied.

All items identical size

Sorting has no effect. FFD still packs ⌊1/s⌋ items per bin correctly. Verifier checks each bin ≤ 1. No special case needed.

Items summing to exactly 2T

Required for Partition. T = (Σaᵢ)/2 > 0 always. Division by T always valid. After normalization Σsᵢ = 2 exactly — backward direction holds unchanged.

Step 3 — Forward Direction (Partition YES → Bin Packing YES)

Recall: T = half the total sum of S (i.e. T = Σaᵢ / 2). Each normalized item has size sᵢ = aᵢ/T.

Lemma 1 — Forward Direction
If ∃ A ⊆ S such that Σi∈A aᵢ = T and Σi∈B aᵢ = T where B = S\A,
∃ A ⊆ S = "there exists a subset A of S" — A is one of the two groups in our partition
Σi∈A aᵢ = T = "the items in group A add up to T" — meaning A is one valid half
B = S\A = "B is everything in S that isn't in A" — the other half
then f(⟨S,T⟩) has a valid packing into k = 2 bins.
f(⟨S,T⟩) = the Bin Packing instance we constructed from the Partition input
k = 2 = we only need 2 bins (one per group)
Proof:
Let sᵢ = aᵢ/T for all i.
sᵢ = the normalized size of item i (original size divided by T so it fits between 0 and 1)
Σi∈A sᵢ = (1/T) · Σi∈A aᵢ = T/T = 1 ≤ capacity.
"Sum of all normalized sizes in group A" = (1/T) × "sum of original sizes in A" = (1/T) × T = 1
Since 1 ≤ 1 (capacity), group A fits perfectly in Bin 1 with no overflow.
Σi∈B sᵢ = T/T = 1 ≤ capacity.
Same argument for group B — it also sums to exactly 1 and fits in Bin 2.
Assign A → Bin 1, B → Bin 2. ∴ Bin Packing = YES. □
∴ means "therefore". □ marks the end of the proof.
In plain English: We designed the reduction so that T divides perfectly into 1. Any group that summed to T in the original problem now sums to T÷T = 1 — and 1 is exactly one bin. So we just drop each group straight into its own bin. The partition we already had becomes the packing automatically — no extra work needed.
A={3,2}, Σ=5=T → sᵢ = {3/5, 2/5} = {0.6, 0.4} → Σ = 1.0 → Bin 1 ✓
B={3,2}, Σ=5=T → sᵢ = {3/5, 2/5} = {0.6, 0.4} → Σ = 1.0 → Bin 2 ✓

Step 4 — Backward Direction (Bin Packing YES → Partition YES)

Recall: T = half the total sum of S. All normalized items sᵢ = aᵢ/T, so Σ all sᵢ = (1/T)·2T = 2. The two bins each have capacity 1.

Lemma 2 — Backward Direction
If f(⟨S,T⟩) has a valid packing into k = 2 bins,
f(⟨S,T⟩) = the Bin Packing instance we built from the Partition input S and T
k = 2 = we are told the packing uses at most 2 bins
then ∃ A ⊆ S such that Σi∈A aᵢ = T and Σi∈S\A aᵢ = T.
∃ A ⊆ S = "there exists a subset A of S" — meaning we can find a valid split
Σi∈A aᵢ = T = "items in group A sum to T" — one equal half
S\A = "everything in S not in A" — the other half also sums to T
Proof: Let A = items in Bin 1, B = items in Bin 2.
Since the packing is valid, every item is assigned to exactly one bin — no item appears in both bins and no item is left out. So {A, B} partitions all n items.
(i) Σi∈A sᵢ ≤ 1   and   (ii) Σi∈B sᵢ ≤ 1
sᵢ = normalized size of item i (= aᵢ/T)
This just says neither bin overflows — each bin holds at most 1 (its capacity).
Σi∈A sᵢ + Σi∈B sᵢ = Σall sᵢ = (1/T)·2T = 2
All items together (Bin 1 + Bin 2) = all items total.
Total of all normalized sizes = (1/T) × total of original sizes = (1/T) × 2T = 2.
So the two bins together must hold exactly 2.
∴ Σi∈A sᵢ = 1   and   Σi∈B sᵢ = 1   (uniquely determined)
Two numbers each ≤ 1 that add up to exactly 2 — both must equal exactly 1. No other possibility.
So both bins are forced to be exactly full.
Denormalizing: Σi∈A aᵢ = T·1 = T   and   Σi∈B aᵢ = T·1 = T. □
Denormalizing = multiply each sᵢ back by T to get original sizes aᵢ.
Each bin's original items sum to T × 1 = T. Two groups each summing to T = a valid partition.
In plain English: After normalizing, all items together sum to exactly 2. We have 2 bins each capped at 1. The only way two numbers that are each ≤ 1 can add up to exactly 2 is if both equal exactly 1 — there is no other possibility. So both bins must be exactly full. Now multiply everything back by T and each bin's items sum to T. That's your two equal groups — the partition falls out automatically from any valid packing.
Why this direction is harder: In the forward direction we built the packing ourselves using the partition we already had. Here we go the other way — we are handed some packing that works, with no idea how it was found, and we need to extract a partition from it. The proof works because the math is so tight that any valid 2-bin packing is automatically a valid partition in disguise — there is no room for a packing that fits the bins but doesn't correspond to a partition.
Bin1=[0.6+0.4]=1.0 × T=5 → Σ aᵢ = 5 = T ✓
Bin2=[0.6+0.4]=1.0 × T=5 → Σ aᵢ = 5 = T ✓
Partition = YES ∎

Conclusion: Partition is NP-complete (Karp, 1972). Lemma 1 + Lemma 2 prove the iff: Partition = YES ⟺ Bin Packing = YES. Since Partition ≤ₚ Bin Packing in O(n) and Bin Packing ∈ NP, Bin Packing is NP-complete. ∎

It proves Bin Packing is at least as hard as Partition. Since Partition has no known polynomial-time solution, neither does Bin Packing — justifying FFD as our practical choice.

🔢 Complexity Derivations — How We Get These Numbers

FFD — Why O(n log n)

Step 1 — Sort: Sorting n items costs O(n log n). This is the dominant cost.
Step 2 — Placement: For each item, find the first fitting bin. Naive scan: O(n) per item = O(n²) total. With a balanced BST: O(log n) per item = O(n log n) total.
Total: O(n log n) with optimized implementation.

First-Fit & Best-Fit — Why O(n²)

No sort step. Items processed in original order.
Placement counting argument:
For each of n items, scan all currently open bins:
 — In the worst case, all bins are scanned before placing.
 — At most n bins can ever be open (one per item).
 — So each placement costs O(n) in the worst case.
n items × O(n) per placement = O(n²) total.

With a BST indexed by remaining capacity: O(log n) per placement → O(n log n) total. But this loses the simplicity that makes FF and BF easy to implement.

Brute Force — Why O(kⁿ)

Each of n items can go into any of k bins. Total assignments = k × k × … (n times) = kⁿ.
n=10, k=4: 4¹⁰ = 1,048,576 assignments.
n=20, k=10: 10²⁰ assignments.
At 10⁹ ops/sec: 3×10¹⁰ years. This is why NP-complete problems need approximations.

⚠ Implementation note: this tool caps Brute Force at 200,000 operations. OPT is guaranteed for n ≤ 12. For larger inputs the result shown is the best found within the cap — not necessarily the true optimal.

The (11/9)·OPT Guarantee

Proven by Johnson (1973), confirmed by Simchi-Levi (1994). FFD uses at most (11/9)·OPT + 6/9 bins.
For large OPT the +6/9 is negligible — essentially within ~22% of optimal.
This bound is tight — specific inputs achieve exactly 11/9 of OPT.

How the ratio is derived (weighting function method): Johnson's proof assigns a weight w(s) to each item based on its size — Large (>1/2), Medium (1/3–1/2), or Small (≤1/3). It then shows the total weight of FFD's bins cannot exceed (11/9) times the total weight of any optimal packing. The weight function is carefully designed so this inequality holds for all possible inputs.

Monotonicity paradox: Greedy algorithms like FFD are not monotone — adding an extra item or increasing a bin's capacity can sometimes reduce the number of bins used, because the new item triggers a different packing arrangement. This counterintuitive behaviour shows why greedy heuristics are inherently "unstable" and why proving approximation bounds requires careful case analysis rather than simple intuition.

Space Complexity — All O(n)

All four algorithms only store:
• n item sizes — O(n)
• Bin assignment per item — O(n)
• Remaining capacity per bin — O(k) ≤ O(n)
No algorithm needs more than linear space.

✅ The Bottom Line — Why FFD is the Right Choice

Bin Packing is NP-complete — no polynomial-time exact algorithm is known. Brute force gives the exact answer but is exponential and impractical for n > 10. FFD runs in O(n log n) and is guaranteed within (11/9)·OPT. For any real-world input, FFD is the right choice.
FFD wins because:
① Fast — O(n log n). For n=1,000,000 items: ~20 million operations. Done in milliseconds.
② Provably close — never more than 22% above optimal. For most real inputs, it hits optimal exactly.
③ Simple — sort once, then scan. Easy to implement, easy to verify, easy to understand.
④ Practical — used in real shipping, cloud allocation, and compiler systems today.
💡 The counterexample where FFD fails (4 bins vs OPT=3) has a ratio of 1.33 — still well within the 1.22 guarantee. Even when FFD is wrong, it is not very wrong.
📌 When to use Brute Force instead: For n ≤ 8 items, Brute Force is perfectly feasible and gives the exact optimal answer. If you can afford it — use it. FFD is the right choice only when n is large enough that exponential time becomes impractical.
